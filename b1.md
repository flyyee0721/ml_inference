## 🟢 必背主干（30 秒能背完，其余都是 reference）

**一句话定调**:B2 是「**B1 诊断出 bound → 拉对应的模型级杠杆**」。四块杠杆各治不同 bound:
- **编译 / 算子融合** → 治 **overhead-bound**(减 kernel launch + Python 派发)+ 部分 memory(融合减 HBM 往返)
- **量化 embedding** → 治 **memory-bound**(数据 1/4,省带宽)
- **量化 dense + Tensor Core** → 治 **compute-bound**(低精度走 Tensor Core)
- **蒸馏** → 直接换更小的模型(三种 bound 一起降,但掉精度)

**5 句金句**:
1. B2 不是"会哪些技术",是 **"先 B1 判 bound,再拉对应杠杆"**——脱离 bound 谈优化是耍流氓。
2. **编译治派发/访存,量化治带宽/算力,蒸馏换模型**;前三个近似无损,**蒸馏唯一主动掉精度**。
3. **量化 ≠ 一定快**:要 Tensor Core MMA 支持 + 维度 8/16 对齐,且 embedding 量化是**省带宽不是走 Tensor Core**。
4. recsys 量化第一杠杆是 **embedding(>99% 体积),用 row-wise INT8**(TorchRec/FBGEMM),不是 dense。
5. **算子融合 ⊂ 编译**(Inductor 自动做);**graph break / 重编译**是 compile 没提速的两大元凶。

**决策树骨架**:
```
模型慢/大
 → B1 判哪种 bound?
    overhead → 编译(减 launch)+ 融合 + CUDA Graphs(reduce-overhead)
    memory   → 量化 embedding(row-wise)+ 融合(减 HBM 往返)
    compute  → 量化 dense + Tensor Core(注意对齐)
 → 还不够 / 要更小 → 蒸馏(换 student,接受掉点)
```

## 0. 心智模型:B2 是 B1 的下游

```
B1 测量 → 判 bound ──┬─ overhead ─→ 编译 / 融合 / CUDA Graphs
                     ├─ memory   ─→ 量化 embedding(row-wise)/ 融合
                     ├─ compute  ─→ 量化 dense + Tensor Core
                     └─ 都想更小  ─→ 蒸馏(换 student)
                          │
                          └─→ 测「体积 / 延迟 / 精度」三角 → 回 B1 验证
```

面试官要听的不是"我会量化会 compile",而是这套纪律:**先归因 bound,再拉对应杠杆,最后用三角指标验证**。这也是 B2 区别于"技术罗列"的 L4 视角。

## 1. 编译（compile / export）

### 1.0 问题定位:为什么要编译
eager 模式逐 op 派发,每个 op 一次 Python→CUDA launch。recsys 小算子多 → **overhead-bound**(接 B1)。编译把多个 op 捕获成图、生成融合 kernel,**减少 launch 次数 + Python 开销**,顺带做算子融合(第 3 节)。

> 📌 关键认知:**kernel fusion 的收益主要是"少搬数据"(中间结果留片上、减 HBM 往返),其次是"少发射"(减 launch);它不提高 GPU 利用率/compute 获益、不减 FLOPs、不让 GEMM 这类 compute-bound kernel 更快。** 即 fusion 是"省 IO/搬运"的优化,不是"省算"的优化。(详见第 3 节;实测见 memory `compile-fusion-precise-facts`)

### 1.1 两条路线先分清(🟢):JIT 求快 vs AOT 求部署

| | `torch.compile` | `torch.export` + AOTInductor |
|---|---|---|
| 时机 | 运行时 **JIT** | 导出期 **AOT** |
| 遇到不可追踪代码 | **graph break**:回退 eager,把编译好的"岛"用 eager 粘起来 | **直接报错**:要求全图(单图),不容忍断点 |
| 产物 | 运行时优化,跟 Python 共存 | 可序列化、可移植、能在**非 Python(C++)**运行的产物 |
| 定位 | 求快、保留 Python 灵活 | 求**部署** |

- **旧路线 TorchScript(trace/script):已不再积极维护 / 弃用**,官方推荐迁到 `torch.export`(Triton inference server issue #8300 就是为这事告警)。

### 1.2 trace vs script(🟡,真题:"区别 + 给个 trace 出错的场景")

| | `torch.jit.trace` | `torch.jit.script` |
|---|---|---|
| 工作方式 | 喂样本输入跑一遍,**录下 op 序列** → 静态图 | 直接**分析源码**(TorchScript 能懂的 Python 子集) |
| 控制流 | ❌ 不能处理数据依赖 if/for——**只录下当次走的那条分支** | ✅ 保留 if/for/while,换输入也对 |
| 非 Tensor 中间值 | 会被当**常量烙死**(int、device 等) | 不会 |
| 需要样本输入 | 要 | 不要 |

- **混用**:可以 script 主模型(含控制流),内部 trace 静态子模块。

> ⚠️ **trace 出错的经典场景**(直接答真题,对应实践卡 **B-1**):
> - forward 里 `if x.sum() > 0:` 这种数据依赖分支 → trace 只保留样本走的那支,换输入**静默走错**。
> - 变长(candidate 数 / 序列长)被当**常量烙死** → 换 batch 形状失配或结果错。
> - 修法:改 `script`,或标 dynamic dim。

**最小反例(trace 把分支烙死)**:
```python
def f(x):
    if x.sum() > 0:      # 数据依赖控制流
        return x * 2
    return x - 2

t = torch.jit.trace(f, torch.ones(3))  # 样本 sum>0,只录下 x*2 这一条分支
t(torch.ones(3))    # [2, 2, 2]    ✅ 正好走对
t(-torch.ones(3))   # [-2, -2, -2] ❌ 仍执行 x*2(分支烙死);正确应是 x-2 = [-3,-3,-3]
# trace 时还会 warn:TracerWarning: Converting a tensor to a Python boolean
#                   might cause the trace to be incorrect
torch.jit.script(f)(-torch.ones(3))    # [-3, -3, -3] ✅ script 保留 if,换输入也对
```

**🟡 拓展:三种 "trace" 全景(都叫 trace,机制完全不同)**

| | `torch.jit.trace` | `torch.fx.symbolic_trace` | `torch.compile`(Dynamo) |
|---|---|---|---|
| 怎么抓 | **执行记录**:喂真 tensor 跑一遍录算子 | **符号执行**:传 **Proxy** 假对象走 forward 录算子 | **字节码解释**:钩 CPython 帧 + **FakeTensor** 传 shape/dtype |
| 数据依赖控制流 | **静默烙死**(换输入算错) | **直接报错**(Proxy `__bool__` 抛 `TraceError`) | **graph break**(降级 eager,不烙死不报错) |
| 输出图 | TorchScript IR | **FX GraphModule**(高层 torch 算子) | FX 图(可多段)→ AOT 降 aten 给 Inductor |
| 主要用途 | 部署(**弃用**) | **图变换/量化**(FX graph mode quant、改图) | **JIT 编译优化** |
| 能处理任意 Python | 否(只录一条路径) | 否(碰数据依赖就错) | **能**(break 当逃生口) |

- **同一个 `if x.sum()>0` 三者反应不同**:jit.trace **静默错** / fx **报错** / Dynamo **graph break(结果永远对)**。这就是 Dynamo 能 compile 任意模型而 trace 不能的根因。
- **FX 是"图的数据结构/IR"**;`fx.symbolic_trace` 和 Dynamo **都产 FX 图**,区别是**怎么产**(Proxy 符号执行 vs bytecode 解释)。⚪ 还有 `make_fx`/proxy_tensor 在 **aten 级**用 FakeTensor 抓图(AOTAutograd 用)。
  - ⚠️ Dynamo **不调用** `fx.symbolic_trace`——它用 PEP 523 钩帧、解释字节码自己建 FX 图(已核实)。"都产 FX 图" ≠ "用同一个 trace 器"。

- ⚪ **`fx.symbolic_trace` 怎么应付控制流**(它没有 graph break,只能绕):
  - 碰控制流分两种:① 条件依赖**被 trace 的 tensor(Proxy)**(`if x.sum()>0`)→ **硬报 `TraceError`**(Proxy `__bool__` 抛错);② 条件依赖**非 trace 的 Python 值**(`if self.training`)→ **只跟一条路径、烙死**(像 jit.trace,不报错但图不完整)。
  - bypass 手段:**`torch.fx.wrap`** 把模块级函数标成 **leaf**(只记 `call_function` 黑盒节点、不进去 trace)→ 救①的报错,但**是"藏成黑盒"非"处理"**(图里看不到内部逻辑);其他:**leaf module**(整个 module 当叶子)、**`concrete_args`**(给具体值把控制流特化烙死)、**重构成 `torch.where`**。`fx.wrap` 须在 trace 前、仅对 global function。两种等价写法:**装饰器** `@torch.fx.wrap` 在定义处,或**调用声明** `torch.fx.wrap("my_func")`(传函数名/函数对象)——后者适合**你不拥有定义**(第三方函数)、加不了装饰器时。

**演进线(为什么现在不写 trace/script 了)**:
```
trace/script(TorchScript,弃用)
   trace 烙死 → 被 Dynamo 的 graph break 解决
   script 保留控制流 → 被 torch.compile(break)/ torch.export(torch.cond/Dim)接班
        ↓
torch.compile(求快,容忍 break) / torch.export+AOTInductor(求部署,全图)
```
> 串起来:**trace 的"烙死"是正确性 bug(B-1)、Dynamo 的"graph break"是性能问题(O-1)——同一个数据依赖控制流,旧路线算错、新路线优雅降级。**

### 1.3 graph break(🟡,torch.compile 核心坑,对应卡 O-1)

- **什么是**:torch.compile 遇到不可追踪代码就**断图**——该处回退 eager,前后编译成两段独立图。断点多 = 优化收益被吃光,甚至跟 eager 一样快。
- **常见触发(完整分类,已核实)**:
  - **A 数据依赖(fundamental,值回流,Dynamo 设计上解不了 → 只能 `torch.cond`/`torch.where` 绕)**:① 数据依赖控制流(`if x.sum()>0`);② 数据依赖循环(循环次数依赖 tensor);③ 直接取数据值(`.item()`/`.data_ptr()`/`.tolist()`/tensor 当 bool/int)。
  - **B 副作用(trace 时真执行)**:④ `print`/`logging`/`warnings`。
  - **C 不支持的构造(支持度问题,随版本改善)**:⑤ 不支持的 Python builtin/函数/部分第三方库;⑥ **custom op 缺 fake/meta kernel**——FakeTensor 追踪算不出该 op 的输出 shape/dtype → 没法传播 → break(或 PT2 编译期报错);修法:`@torch.library.register_fake`(旧名 `impl_abstract`)注册一个**只返回正确 shape/dtype、不做真实计算**的假实现,op 就能留图不 break。⑦ 不支持的 context manager(如 `with open(...)`);⑧(历史)try/except、generator、closure 等(新版很多已支持)。
  - 区分:**A 类是根因性的(值回流,绕不开只能改写)**;**B/C 类是 Dynamo 支持度问题**。
- **定位**(⚪):`TORCH_LOGS=graph_breaks`;`torch._dynamo.explain`;`fullgraph=True` 让它在断图处**直接报错**(强制无断点)。
- **修 `.item()`**:`torch._dynamo.config.capture_scalar_outputs=True`(或 env `TORCHDYNAMO_CAPTURE_SCALAR_OUTPUTS=1`);或把 `.item()`+Python if 换成 tensor 运算(`torch.where` / `torch.cond`)。
- 🪖 **这正是那张战绩卡(O-1)的雏形**:compile 跑了却没加速 → `TORCH_LOGS=graph_breaks` 一看是 `.item()` 触发 → 挪出 forward / 换 tensor 运算 → 提速。

> ⚠️ **循环里的 graph break = 整帧 eager fallback(灾难级,实测踩到)**:break 不总是"切两段还能各编各的"。**当 break 发生在 for/while 循环体内,Dynamo 不支持"循环里断图",会直接 skip 整个 frame、整个 forward 退回 eager**(报 `graph break in loop ... falling back to eager`)。后果:`torch.compile` 等于**完全没生效**,一点加速都没有。
> - 实测佐证:用自定义 `backend` 想打印 FX 图时**什么都没打印**——因为整帧退 eager、**根本没有图传给 backend**(backend 没被调用)。"图打不出来"本身就是"全 eager"的铁证。
> - 启示:recsys 的逐层/逐 candidate 循环里若有 `.item()`/数据依赖 if,危害被**整帧 eager** 放大,不是"少融一点"。优先把循环内的数据依赖消掉(`torch.where`/`torch.cond`)。

#### 🟡 control flow 和 graph break 完整总结(为什么会 break 的根因)

**根本原因:Python control flow 需要值回流**

Dynamo 追踪时只记录算子序列，不关心 tensor 里的数据内容。但 Python 的 control flow 语句（`if`/`while`/`for`）在语言语义上要求把条件求值成 Python `bool` 或 `int`——当条件是 tensor 时，Python 隐式调用 `tensor.__bool__()` 或 `tensor.__index__()`，把 GPU tensor 的值拉回 Python 层。这个回流是 Python 语言本身的规定，不是 PyTorch 能绕过的。

Dynamo 遇到回流之后必须知道这个值是什么，才能决定追踪哪条代码路径——因为 Dynamo 不支持 bailout，不能同时保留多条 branch，必须在 tracing 时静态确定走哪边。知道不了就只能 graph break。

**三种 control flow 的具体情况：**

**`if`：**

```python
if mask.any():      # Python 要求 bool
                    # mask.any().__bool__() → 值回流
                    # Dynamo 必须知道 True/False
                    # unbacked 时不知道 → break
    return x * 2   # 这条 FX graph
return x * 3       # 和这条完全不同，只能选一条
```

backed 且值已知时，Dynamo 特化到某个 branch，生成 strict equality guard，条件变化时 guard 失败重编译。unbacked 时不知道走哪边，break。

**`while`：**

```python
while mask.any():   # 每次循环都要 __bool__() 回流
    x = x * 2      # 循环次数也不知道
                    # 比 if 更严重，不只是"走哪边"
                    # 还有"循环几次"的问题
                    # → break
```

**`for`：**

```python
for i in range(10):    # range(10) 是纯 Python 对象
                       # 10 是编译时常数
                       # Dynamo 直接展开，不 break

for i in range(n):     # n 是 dynamic/unbacked
                       # 循环次数不知道 → break

for row in tensor:     # 迭代 tensor 本身
                       # 行数依赖 shape
                       # dynamic shape 时 → break
```

**`torch.where` 不 break 的原因：**

```python
result = torch.where(condition, x, zeros)
# Dynamo 看到的：
# %condition = aten.gt(%x, 0)         ← GPU op，输出 bool tensor
# %result = aten.where(%condition, %x, %zeros)  ← GPU op
```

没有 Python `if`，没有 `__bool__()`，condition tensor 里的值完全不需要回流到 Python。判断和选择在 GPU kernel 里完成，Python 层只是"调用函数、传入参数、得到返回值"。Dynamo 只记录两个算子，不需要知道 condition 的内容，不影响任何 Python 代码路径，不 break。

**完整规律：**

```
Python control flow + tensor 条件
  → tensor.__bool__()/__index__() 回流
  → Dynamo 必须知道具体值
  → unbacked/推导不出来 → graph break
  → backed/能推导 → 特化到某个 branch + guard

torch.where / torch.masked_select 等 tensor 算子
  → 没有 Python control flow
  → condition 留在 tensor 世界
  → Dynamo 只记录算子，不需要知道内容
  → 不 break

for range(常数)
  → 纯 Python 对象，编译时已知
  → Dynamo 展开循环
  → 不 break
```

**和 shape hint 的联系：**

`torch._check` 解决的正是"backed/能推导"这个条件——给 ShapeEnv 加约束，让 SymPy 能从约束推导出控制流的结果，不需要真正回流就能确定走哪个 branch：

```python
n = x.nonzero().shape[0]   # u0，unbacked
torch._check(n >= 1)        # ShapeEnv: u0 >= 1

if n > 0:   # SymPy 推导：u0 >= 1 → u0 > 0 永远 True
            # 不需要回流，直接确定走 True branch
            # 不 break
```

本质上 `torch._check` 是在告诉 Dynamo："你不需要真正回流这个值，我保证它满足这个条件，你直接按这个条件走就行。"

### 1.4 编译栈三段(🟡):捕获 → 反向 → codegen

`torch.compile` 不是单体,是一条流水线(知道分段才能定位"卡在哪一段"):

| 段 | 组件 | 干什么 |
|---|---|---|
| ① 图捕获 | **TorchDynamo** | 钩进 CPython 帧求值 API(PEP 523),**改写 Python 字节码**,把 PyTorch op 序列抽成 FX 图;遇不可追踪就 **graph break**(1.3) |
| ② 反向图 | **AOTAutograd** | 由前向图 AOT 生成**反向图**(训练才需要);之后进入"纯计算图"世界 |
| ③ 代码生成 | **TorchInductor**(默认后端) | 把图 lower 成 kernel:**GPU→Triton**(shared memory/warp 并行)、**CPU→C++/OpenMP**(多线程+AVX2) |

- Inductor 是**算子融合**的落地处(第 3 节):多个 op 融成一个生成 kernel,减少 HBM 往返。
- 为什么要懂分段:graph break 是 ① Dynamo 的事;没融合/没提速是 ③ Inductor 的事;重编译是 guards 的事(下一节)。

> ⚪ **三个 IR + decomposition vs lowering(术语要分清,已核实)**:PT2 栈有**三种 IR**——**Torch IR**(Dynamo 产,高层算子如 `nn.Linear`/`F.gelu`)→ **ATen IR**(AOTAutograd 产,基本算子如 `aten.addmm`/`aten.erf`)→ **Loop-level IR**(Inductor 产,逐元素循环级)。两步两个名字:
> - **decomposition(分解)= AOTAutograd 做**:把 Torch IR 分解成 ATen/Prims IR(`nn.Linear→aten.addmm`、`F.gelu→aten.mul+erf+add…`),同时 **functionalization**(把 mutation/view 改写成函数式)。
> - **lowering(降级)= Inductor 做**:把 **ATen op → Inductor Loop-level IR**(Pointwise/Reduction/ComputedBuffer…),再 codegen 成 Triton/C++ kernel。
> - ⚠️ 易错:"lowering"**特指 ATen→Loop IR 这一步**,不是 Torch→ATen(那叫 decomposition)。
> - 关键认知:lowering **不是改写现成 ATen kernel**(eager 的 cuDNN/cuBLAS),而是把算子降成"循环"形式,好让 Inductor **跨算子融合 + 生成全新定制 kernel**。
> - 链条:`Torch IR →[decomposition+functionalization,AOTAutograd]→ ATen IR →[lowering,Inductor]→ Loop IR →[codegen]→ Triton/C++`。

> 🪖 **调试 tip:`torch.compile(model, backend="eager")` 只走 ①②、跳过 ③**——它**捕获/生成图但不做算子融合等 Inductor 优化**,图直接交回 eager 执行。所以想**单看 Dynamo 抓出来的图、排查 graph break/重编译**(而不被 Inductor 的融合/codegen 干扰)时,用 `backend="eager"` 最干净;`backend="aot_eager"` 则多走 ② 反向、看带反向的图。

> 🪖 **自定义 backend 截获 FX 图(亲手看图最稳的方式)**:backend 就是个"拿到 GraphModule 后返回可调用对象"的函数——在里面打印图,再原样返回执行:
> ```python
> import torch
> def show_graph(gm, example_inputs):   # gm: 抓到的 GraphModule;example_inputs: 例子输入
>     print("====== FX GRAPH ======")
>     print(gm.graph)                    # 最原始,绝对能打(opcode/target/args)
>     gm.print_readable()                # 可读源码版:def forward(...): linear=...; relu=...
>     return gm.forward                  # ★ 必须返回可调用对象,否则 'NoneType' not callable
> _ = torch.compile(model, backend=show_graph)(x)   # 用 _ 接住返回值,免得 notebook 再显示输出 tensor
> ```
> ⚠️ 两个实测坑:① backend **不 `return gm.forward` → 调用时报 `NoneType not callable`**;② **有 graph break 时 backend 被调多次**(每段子图一次);**整帧退 eager 时 backend 根本不被调用 → 什么都不打印**(见 §1.3 循环 break)。改名/换模型后若没打印,先 `torch._dynamo.reset()` 清缓存(否则命中缓存、backend 不触发)。

> ⚪ **`torch._logging.set_logs(...)` 常用 param 速查**(= 环境变量 `TORCH_LOGS="..."`,逗号分隔):
>
> | param | 打印什么 |
> |---|---|
> | `graph` | Dynamo 抓的 **FX 图**(节点) |
> | `graph_code` | Dynamo 图的 **Python 代码**形式(最易读) |
> | `output_code` | **Inductor 生成的 kernel**(Triton/C++)——看融合 |
> | `graph_breaks` | 每个**唯一 graph break** 的消息(断在哪、为什么) |
> | `recompiles` | **为什么重编译**(哪个 guard 变了) |
> | `guards` | 每个编译函数**生成的 guards** |
> | `dynamic` | **动态形状/符号推理**:guard/符号尺寸何时为何被加 |
> | `aot_graphs` / `aot_joint_graph` | **AOTAutograd** 的前/反向(及 joint)图 |
> | 组件级:`dynamo` / `inductor` / `aot` / `dynamic`(设 `logging.DEBUG`) | 整个组件的详细日志 |
>
> 两种设法:① `torch._logging.set_logs(graph_breaks=True, output_code=True)`(运行时,**import 后即可**);② `TORCH_LOGS="graph_breaks,output_code"` 环境变量(**须在 `import torch` 之前设**,否则不生效——实测 gotcha)。

### 1.5 guards 与重编译(🟡,动态形状,对应 B3 §4.6 / 卡 B-3)

- **guard 是什么**:编译产物装上一组**前置条件**(输入形状、dtype、值等),复用前先逐条校验;**guard 失效 → 重编译**生成新图。
- **形状一变就重编译**:recsys 的 candidate 数 / 序列长每请求都变 → 默认每遇新形状重编译一次,**重编译风暴 → 尾延迟尖刺**(这正是 B3 §4.6 的坑、实践卡 B-3)。
- **三种应对**(⚪):
  - 默认(`dynamic=None`):遇新动态张量就重编译。
  - **`torch._dynamo.mark_dynamic(tensor, dim)`**:提前标某维动态,让它编一张尽量通用的图,避免反复重编译(可给 min/max)。
  - `dynamic=False`:永不生成动态图,每个新形状各编一张静态图进缓存。
- **定位**(⚪):`TORCH_LOGS=recompiles` 看重编译触发;`TORCH_LOGS=dynamic` 看 guard 何时/为何被加(诊断过度特化)。

### 1.5.1 应对 dynamic shape 的策略梯度(recsys 变长命门)

> 面试问"recsys 变长(candidate 数/seq len)怎么配 torch.compile"——要能讲清**机制(为什么)→ 三种 symint → 手段梯度**。本节 🟡 是机制、🟢 是速查。

#### 🟡 机制:Dynamo 在 tracing 时怎么处理不确定的 shape

`torch.compile` 是 JIT,**第一次真正 call 才触发 tracing**。Dynamo 用 **FakeTensor** 追踪——FakeTensor 有真实 input 的 **shape/dtype/device,但没有真实数据**。所以:
- **input 的 shape** → Dynamo 知道(**backed**)
- **数据内容决定的 shape** → Dynamo 不知道(**unbacked**)

**三种 SymInt 对比**:

| | 来源 | 编译期 | kernel 里 | guard / 重编译 |
|---|---|---|---|---|
| **常量** | 编译时已知具体值 | hardcode | 常量 | 静态优化全开(展开/折叠/针对 size 的 autotune) |
| **backed `s0`** | input shape(有 hint value=首次 call 值) | 默认 **specialize** 成 hint 值 | 默认是常量;`mark_dynamic` 后是**参数** | 默认 **equality guard**(`shape==10`,变了重编译);`mark_dynamic` 后 **range guard**(`1<=s0<=100`,范围内不重编译,但静态优化受限) |
| **unbacked `u0`** | 数据内容(`nonzero`/`unique`/`item`/`mask` 索引) | **无 hint、无 guard** | **天然是参数** | **天然不重编译** |

#### 🟡 graph break 的根本原因

Dynamo **不支持运行时 bailout**(graph break 本身就是它的 bailout——在 **trace 期**断图回退 eager,而非运行时回退),所以**必须在 trace 时确定走哪个 branch**:
```python
n = x.nonzero().shape[0]   # u0,unbacked
if n > 0:                  # 不知 True/False → graph break
    ...
```
- **unbacked 在控制流里 → 不知走哪边 → graph break**(fullgraph 下报 `GuardOnDataDependentSymNode`)。
- ⚠️ 对照:**backed 在控制流里 ≠ break**——它有 hint value,**specialize 到 hint 那支 + 加 guard**(分支变才重编译),不 break。
- graph break 把图切多段,跨 break 的 fusion 做不了(接 §1.3)。

#### 🟡 torch._check vs mark_dynamic 的精确分工

- **`torch._check(n >= 1)`**:给 **ShapeEnv**(Dynamo 内部的 SymPy 推导环境)**加约束**。SymPy 用它推控制流——`u0>=1` → `u0>0` 永远 True → 不需运行时判断 → **不 break**。**解的是 graph break,不解重编译**(kernel 里 u0 还是参数);运行时插 assert 验证,违反直接 **RuntimeError(不是重编译)**。
- **`mark_dynamic(t, dim, min, max)`**:告诉 Dynamo 这维动态、别 specialize → 生成 **range guard** 替代 equality guard → **解重编译**;min/max 同时给 ShapeEnv 加约束 → **隐式解 break**。⚠️ 不能在被 compile 的函数(forward)内部调;只对 tensor 维、不对 int。
- 组合:**`torch._check` 解 break + `mark_dynamic` 解重编译 → 既不 break 也不重编译**。
- ⚪ 进阶:`nonzero/unique` 的 unbacked 常被当 **size-like(假设 ≥0)**,有 `torch._check_is_size()`/`mark_unbacked`;`.item()` 要 `capture_scalar_outputs`。

#### 🟡 mark_dynamic vs maybe_mark_dynamic + constraint violation 怎么修

**两者区别(面试常问)**:

| | `mark_dynamic` | `maybe_mark_dynamic` |
|---|---|---|
| 语义 | **断言**这维动态 | **尽量**动态 |
| 被某操作逼特化时 | **报 constraint violation 错** | **静默退回特化**(不报错) |

> ⚠️ `maybe_mark_dynamic` **消报错但不消特化**——被逼特化时它退回 static、**照样可能重编译,只是不告诉你**。`mark_dynamic` 那个报错其实是**有用信号**:在喊"某行把这维逼死了,去修它"。想**真动态**别用 maybe 静音,去修逼特化的那行。

**什么操作会逼特化(→ constraint violation)**:凡需"知道这维确切值才能往下走"的——`if x.shape[0]==64`(相等控制流)、`x.view(64,-1)`(写死)、把 shape 当 python int 查表/控制流。

**解决决策树**:
```
constraint violation
 → 写死的常量?      → 改用符号:x.view(64,-1) → x.view(x.shape[0],-1)   (首选,解决大半)
 → shape 控制流?    → 能删就删 / 用 torch.cond(functional 控制流,进图不 break)
                       / 换 mask·torch.where(像 fixed 模型把 .item()+if 换 where)
 → 证明不了的约束?  → torch._check(x.shape[0] % 8 == 0) 补事实
 → 本质必须静态?    → 别 mark_dynamic,接受重编译 / padding·bucketing(§梯度第1条)
```

`torch.cond` 用法(替代 python if,避免特化/break):
```python
# ❌ y = big(x) if x.shape[0] > 100 else small(x)   # 逼特化/break
y = torch.cond(x.shape[0] > 100, big, small, (x,))   # ✅ 走哪支变成图里算子,运行时定
```

#### 🟡 shape 对 graph / kernel 的影响

- **FX graph**:shape **不影响控制流** → 拓扑相同,只是 annotation 从具体整数变成 SymInt 表达式(`s0`、`s0+1`、`2*s0`);shape **影响控制流** → 拓扑可能完全不同(static 一支 / dynamic 可能 break 成多段)——**这是差异最大处,也是 shape hint 最重要的作用**。
- **kernel**:static(常量)→ hardcode、循环展开、最优 autotune、最优 memory planning;dynamic(参数)→ 循环次数运行时才知、不能展开、autotune 选"整个 range 都还行"的配置、memory planning 保守 → **性能略差但不重编译**。

#### 🟢 速查:策略梯度(从最稳到最灵活)

1. **padding / bucketing 到少数固定 shape(最稳)**:compile 只见有限 shape → 零数据依赖、零重编译。代价 padding 浪费(接 B3 §4.5)。
2. **`mark_dynamic(t, dim, min, max)`**:backed 的 hint,一上来编通用图,绕开"特化→变 shape→重编译"。
3. **automatic dynamic(`dynamic=None` 默认)**:首次静态、recompile 时自动转 dynamic → 省心但**白付第一次重编译**(serving 首请求尖刺)。
4. **`dynamic=True/False`**:True 强制全动态(不重编译、略慢);False 每个新 shape 各编一份。
5. **`torch._check(约束)`**:unbacked(数据依赖尺寸如 candidate 数)的解药——编译期化解数据依赖报错 + 运行时断言,**不涉重编译**。
6. **`torch.export` 的 `Dim(name, min, max)` + `dynamic_shapes=`**:部署给 AOTInductor 时声明动态维(或 `Dim.AUTO/DYNAMIC/STATIC`)。

**定位**(⚪,排查梯度由粗到细):
- `TORCH_LOGS=recompiles` 看重编译触发;`TORCH_LOGS=dynamic` 看 guard/符号尺寸为何被加(知道**发生了**)。
- `TORCHDYNAMO_EXTENDED_DEBUG_GUARD_ADDED="<guard>"` / `TORCHDYNAMO_EXTENDED_DEBUG_CREATE_SYMBOL="s0"`:某个**特定 guard 被加 / 某个符号被创建**时打完整栈,揪出具体元凶在代码哪行产生(定位到**谁干的**)。
- `TORCHDYNAMO_EXTENDED_DEBUG_CPP=1`:guard 疑似从 **C++ 层**触发时补 C++ 栈。

#### 🪖 实操:用 `graph_code` 在 FX 图里看 symint / torch._check(亲手验证机制)

看编译后的图/kernel 的 API:`TORCH_LOGS=graph_code`(看 FX 图对应代码)/ `output_code`(看生成的 Triton/C++ kernel)/ `recompiles`。

实测 `.item()` + `capture_scalar_outputs=True` + `torch._check(s>0)`,`graph_code` 打出:
```
gt_78: "Sym(zuf39 > 0.0)" = item_39 > 0          # .item() 进图,不 break
_check_39 = torch._check(gt_78)                  # torch._check 被编进图:注册约束给 ShapeEnv
```
读出三件事:① `.item()` 被捕获进图(没 break,因为开了 `capture_scalar_outputs`);② `zuf39` 是这个值的 **unbacked 符号**;③ `s>0` 成了**符号布尔** `Sym(zuf39>0)`,`torch._check` 把"假设它成立"写进图(编译期假设)。

**看符号前缀就知道是哪类**(排查动态形状必备):
- **`s0, s1…`** = backed symint(input shape,有 hint)
- **`u0, u1…`** = unbacked symint(`.shape[0]`/`nonzero`,无 hint)
- **`zuf…`** = unbacked symfloat(`.item()` 的浮点值)

#### 🟢 重编译触发条件(速记)
```
guard 失败(backed s0 超出 guard 范围) → 重编译
torch._check 违反                      → RuntimeError(非重编译)
unbacked u0 变化                       → 通常不重编译:
   不在控制流 → 正常,u0 作参数传入
   在控制流 + 有 check → check 解析分支(违反则运行时报错)
   在控制流 + 无 check → graph break,eager 执行
   极少数影响了 backed guard → guard 失败,重编译
```

> 30 秒答法:recsys 变长配 compile,先分 symint——**backed(普通变长)** 用 `mark_dynamic` 给 min/max 编通用图、或默认 automatic dynamic(白付一次重编译)、最稳 **padding/bucketing**;**unbacked(数据依赖如 candidate 数)** 用 **`torch._check`** 补 ShapeEnv 约束化解 graph break(编译期假设+运行时断言,不涉重编译)。机制根因:Dynamo 用 FakeTensor 无数据、trace 期必须定 branch,所以 unbacked 进控制流就 break。部署走 export 用 `Dim(min,max)`。

> **本节事实来源**:[FakeTensor / backed·unbacked / 动态形状核心概念](https://docs.pytorch.org/docs/main/user_guide/torch_compiler/compile/dynamic_shapes_core_concepts.html)、[torch._check + ShapeEnv + GuardOnDataDependentSymNode 排查](https://docs.pytorch.org/docs/main/user_guide/torch_compiler/compile/dynamic_shapes_troubleshooting_guardon_errors.html)、[mark_dynamic min/max + automatic dynamic + dynamic flag](http://docs.pytorch.org/docs/main/user_guide/torch_compiler/torch.compiler_dynamic_shapes.html)、[torch.export Dim(name,min,max) + dynamic_shapes](https://docs.pytorch.org/docs/main/user_guide/torch_compiler/export.html)。

### 1.6 mode 选项(🟡):default / reduce-overhead / max-autotune

`torch.compile(model, mode=...)` 三档,核心差异是**编译时间 ↔ 运行性能**:

| mode | 做什么 | 代价 |
|---|---|---|
| `default` | 常规融合 + codegen | 编译快 |
| **`reduce-overhead`** | 在上面再叠 **CUDA Graphs**:把多个 GPU kernel 录成可重放序列,**消掉每个 kernel 的 CPU launch 开销** | 编译变慢;吃显存 |
| `max-autotune` | 更激进地 autotune kernel(也默认开 CUDA Graphs),追求最快运行 | 编译**最慢** |

- **接 B1**:`reduce-overhead` 的 **CUDA Graphs** 正是治 **overhead-bound**(小 batch / 多小 kernel,CPU launch 成瓶颈)的杠杆——recsys 小算子多,这档常有效。
- ⚠️ 但 CUDA Graphs 对**动态形状不友好**(形状变要重新实例化 graph),和 1.5 的重编译叠加可能**更慢**——开之前实测。

### 1.7 AOTInductor(🟡,部署新路径,对应工业案例)

- **用途**:把 `torch.export` 出来的图**编译成 shared library**,用于**非 Python(服务端 C++)推理部署**。
- **API 三步**(⚪):
  1. `torch.export.export()` —— 捕获计算图
  2. `torch._inductor.aoti_compile_and_package()` —— 编译 + 打包成一个 .pt2 包
  3. 加载:Python 用 `torch._inductor.aoti_load_package()`;**C++ 用 `torch::inductor::AOTIModelPackageLoader`**
- **为什么重要**:它是 TorchScript 之后的**官方 C++ 部署路径**——"trace/script 不再是终点"(接实践 plan 📖 工业案例)。

### 1.8 torch.compile vs torch.export 一句话收口(🟢)

- **compile** = 运行时 JIT、容忍 graph break、与 Python 共存,**求快**。
- **export** = 导出期全图、不容忍断点(报错)、可序列化移植,**求部署**(配 AOTInductor 落到 C++)。
- 注:export 也有 strict / non-strict 模式;non-strict 有时比 compile 的 fullgraph 更容易跑通。

### 1.9 面试答法(30 秒)

> 编译是治 recsys **overhead-bound** 的主杠杆——eager 逐 op 派发、小算子多,编译捕获成图减 launch + 融合 kernel。栈分三段:**Dynamo** 改字节码抽 FX 图(graph break 发生在这)→ **AOTAutograd** 生成反向图 → **Inductor** 生成 kernel(GPU 出 Triton、CPU 出 C++/OpenMP)。两条路线分清:`torch.compile` 是运行时 JIT,遇不可追踪代码 **graph break** 回退 eager(`.item()`/数据依赖 if 是常见元凶,`TORCH_LOGS=graph_breaks` 定位);它靠 **guards** 复用,形状变就**重编译**(recsys 变长易触发,`mark_dynamic` 缓解),`mode=reduce-overhead` 还能叠 **CUDA Graphs** 进一步消 launch 开销。部署侧 `torch.export`+**AOTInductor** 导出全图、产出能在 **C++** 跑的 shared lib,是 TorchScript(已弃用)之后的官方路径。老 `trace` 不能处理数据依赖控制流、会把分支和非 Tensor 值烙死——带控制流要 `script`。

> **本节事实来源**:[trace vs script](https://discuss.pytorch.org/t/when-should-i-use-tracing-rather-than-scripting/53883)、[Common Graph Breaks (PyTorch docs)](https://docs.pytorch.org/docs/stable/user_guide/torch_compiler/compile/programming_model.common_graph_breaks.html)、[编译栈 Dynamo/AOTAutograd/Inductor](https://docs.pytorch.org/docs/stable/torch.compiler_dynamo_overview.html)、[Inductor codegen (PyTorch 2.x)](https://pytorch.org/get-started/pytorch-2-x/)、[guards/重编译与动态形状](https://docs.pytorch.org/docs/main/user_guide/torch_compiler/compile/dynamic_shapes_core_concepts.html)、[mode reduce-overhead/max-autotune + CUDA Graphs](https://towardsdatascience.com/maximizing-ai-ml-model-performance-with-pytorch-compilation/)、[AOTInductor (PyTorch docs)](https://docs.pytorch.org/docs/main/user_guide/torch_compiler/torch.compiler_aot_inductor.html)、[torch.compile vs torch.export](https://dev.to/minwook/pytorch-compile-vs-export-omc)、[TorchScript 弃用 / Triton issue #8300](https://github.com/triton-inference-server/server/issues/8300)。

## 1.10 部署路线全景:在线(JIT)vs 离线(AOT)(🟡 综合)

> 把前面散点(compile / export / AOTInductor / TorchScript)收成一张部署地图。**核心结论:在线(JIT)和离线(AOT)用的是同一个 Inductor、优化能力相同,区别只在触发时机、是否需要 Python、能否序列化分发。**

### 核心区别

```
在线(JIT):运行时第一次 call 编译;需要 Python;结果绑进程
离线(AOT):部署前编译;产物可序列化分发;运行时不需要 Python
```

### 在线:torch.compile(JIT Inductor)

```
第一次 call(x) → Dynamo 追踪(FakeTensor,有 shape/dtype/device)
  → FX 图(torch 级,宽松,允许 graph break)→[AOTAutograd 降 ATen]→ Inductor(fusion/autotune/codegen)
  → 进程内 Triton/CUDA kernel → 后续 call 复用
```
- **Guard**:首次编译的假设变 guard;后续 call 校验,通过则复用、失败则重编译。
- **Graph break**:遇不支持的就 break 多段、回退 eager,**结果永远对、只损性能**。
- **Dynamic shape**:`mark_dynamic` + `torch._check` → range guard,范围内不重编译。
- ⚠️ **缓存(纠正"进程结束消失")**:torch.compile **能持久化缓存**——`FxGraphCache`/`AOTAutogradCache` 落盘**跨进程复用**;`Mega-Cache`(`torch.compiler.save_cache_artifacts()`/`load_cache_artifacts()`)**可移植、搬到另一台机器预热**。**但这是"避免重编译"的缓存,不是无 Python 部署产物**——用它仍需 Python/torch。
- **适用**:训练加速、Python 推理(能接受/可预热 cold start)、研究迭代、dynamic shape 场景。

### 离线:torch.export + AOTInductor

```
部署前(Python 里):torch.export(model, args, dynamic_shapes=...)
  → ExportedProgram(严格 ATen IR,不允许 break/副作用,dynamic 显式声明)
  → AOT Inductor(同一个 Inductor)→ model.so(CUDA kernel + C++ wrapper)
运行时(C++,无 Python):dlopen("model.so")→ 直接调 kernel
```
- 严格全图(AOT 不能有 fallback);**无解释器/Python 开销**;`.so` **可序列化分发**(打包进 Docker / 边缘设备);**编译开销在离线**,生产首 request 不等编译。
- **适用**:C++ serving(核心)、延迟敏感不能 cold start、边缘/嵌入式无 Python、shape 相对稳定。

### TorchScript(旧离线方案)

```
trace/script → ScriptModule(.pt) → torch::jit::load → LibTorch 解释执行 TorchScript IR
```
- ⚠️ **纠正:TorchScript 有 fusion(不是无)**——`NNC(TensorExpr)`/`nvFuser` 做 **pointwise 融合**;但**有限**(主要 pointwise、runtime),弱于 Inductor 的"pointwise+reduction+template 全融合 + 完整 Triton/C++ codegen + autotune"。
- 区别于 AOTInductor:**有 LibTorch 解释器开销、fusion 弱、无自定义 codegen kernel**;但成熟度高、已有代码可继续用。TorchScript 已弃用,新项目 C++ 部署首选 AOTInductor。

### 同一个 Inductor,两种触发

| | torch.compile(JIT) | AOTInductor(AOT) |
|---|---|---|
| Inductor / fusion / autotune / codegen | **同一个,能力相同** | **同一个,能力相同** |
| 输入图 | Dynamo 内部图(宽松,允许 break) | ExportedProgram(严格、完整、无副作用) |
| 触发 | 运行时首次 call | 离线、部署前 |
| 输出 | 进程内 kernel(可缓存预热,见上) | `.so`,可分发 |
| Python | **需要** | **不需要** |
| cold start | 有(可用 Mega-Cache 预热消) | 无 |

### 场景选择速查

| 场景 | 选 |
|---|---|
| 训练加速 / Python 推理 / 研究迭代 / shape 频繁变 | **torch.compile**(+ `mark_dynamic`) |
| C++ serving / 延迟敏感不能 cold start / 边缘无 Python / shape 固定 | **torch.export + AOTInductor** |
| 已有 TorchScript 代码 | 继续用,不必迁 |
| 移动端 | torch.export + **ExecuTorch** |
| 跨框架(非 PyTorch 生态) | **ONNX** + TensorRT/ONNX Runtime |

### 一张图收口

```
PyTorch 模型(eager)
 ├─ torch.compile ───────────────→ JIT kernel(在线,需 Python,可缓存预热)
 ├─ torch.export → ExportedProgram
 │       ├─ AOTInductor → .so ───→ C++ 直接调(离线,无 Python)
 │       ├─ ExecuTorch ─────────→ 移动端
 │       └─ dynamo onnx → .onnx ─→ TensorRT / ONNX Runtime
 └─ jit.trace/script → .pt → LibTorch → C++(旧路线,解释器开销,弱 fusion)
```

## 2. 量化（quantization）

### 2.0 问题定位:recsys 为什么"优先量化 embedding"(🟢)

- **DLRM 类推荐模型 >99% 的体积是 embedding 表**,工业级可达 **TB ~ 几十 TB**。dense MLP 只占零头。
- 所以 recsys 量化的第一杠杆是 **embedding 表**(省体积 + 省带宽,治 **memory-bound**),不是 dense。这是 recsys 区别于 LLM(LLM 大头在 dense 权重)的关键。
- 代价:embedding 直接关系模型质量,量化掉点风险高 → 量化方式选择尤其重要(2.2/2.3)。

### 2.1 量化是什么(🟡)
把 float 映射到低位整数(INT8/INT4):`q = round(x/scale) + zero_point`,用时再反量化。两个参数:**scale**(缩放)、**zero_point**(零点偏移)。省体积、省带宽,低精度还能走 Tensor Core(2.4)。

### 2.2 PTQ vs QAT(🟡)

| | **PTQ**(训练后量化) | **QAT**(量化感知训练) |
|---|---|---|
| 怎么做 | 训练完直接量化(拿小校准集定 scale) | 训练时就**模拟量化**(前后向插 fake-quant),让模型学会适应量化误差 |
| 精度 | 中等位宽够用;**激进位宽(2/4bit)易崩** | 通常更准(约比 PTQ 高 ~10%),激进位宽能救回 |
| 成本 | **快几个数量级**(分钟级,无需重训) | 贵(要重训) |
| recsys 用法 | embedding **8-bit** 的好起点 | embedding 压到 **INT4** 时上 QAT,可达约 8× 体积压缩 |

> 决策:**先 PTQ 试,掉点超预算再上 QAT**。8-bit 多数 PTQ 够;4-bit 往下考虑 QAT。

**PTQ 内部还分 dynamic vs static(激活怎么量化,面试常追问)**:

| | **dynamic** 动态 | **static** 静态 |
|---|---|---|
| 激活 scale | **运行时**按每 batch 现算 | **校准集预先**定好(需 calibration 步骤) |
| 要校准集吗 | 不要,即插即用 | 要,有代表性的校准数据 |
| 速度 | 有运行时开销,提速有限 | **全程 INT8 算,最快**;省内存最多 |
| 精度 | 每输入精确算 range,常更稳 | 校准没覆盖的 outlier 会被 clip → 掉点 |
| 适用 | CPU 推理快速起步 | 输入分布已知、追求最大加速(GPU 部署主选) |

> 权重总是离线量化;dynamic/static 的区别只在**激活**。

### 2.3 粒度与对称性(🟡,真题"代价是什么"答这)

**粒度:per-tensor vs per-channel**
- per-tensor:整张张量**一组** scale/zp。便宜,但**各 channel 动态范围差异大时会被拉崩**(一个 outlier channel 撑大 range,其他 channel 精度被牺牲)。
- per-channel:每个 channel(权重输出通道 / 激活每 token)**各自**一组 scale/zp。**更准**,但元数据和计算开销大。LLM/recsys 权重量化常用 per-channel。

**对称 vs 非对称**
- 对称:range 以 0 为中心,**无 zero_point**。计算便宜(少一步偏移)。
- 非对称:用 zero_point 把 float 的 min/max 精确映射到 int min/max。**更准**(尤其分布不对称、如 ReLU 后全正),但反量化多一步算。

> 🪖 对应卡 **B-2**:对有 outlier 的 embedding 用 **per-tensor 对称** → 精度崩;换 **row-wise / 非对称** → 回正。这就是"量化方式选错"的根因。

### 2.3.1 recsys 落地:embedding 用 row-wise 量化(🟢,实践 Demo B 用这个)

> ⚠️ 面试问"embedding 怎么量化"答 per-channel 是**不精准**的。embedding 表的实际做法是 **row-wise(per-row,每行一组 scale/zp)**——每个 embedding 向量(表的一行)独立量化。

- **为什么 row-wise**:embedding 不同 id 的行幅度差异大,整表一组 scale(per-tensor)会被热门行的大值拉崩;每行一组 scale 精度好得多,且 lookup 时只取用到的行、scale 跟着行走,开销可控。
- **工具(已核实)**:**TorchRec** 提供 `QuantEmbeddingBagCollection` / `QuantEmbeddingCollection`(INT8 权重量化),底层 **FBGEMM** kernel 支持 **row-wise / tensor-wise / column-wise / block-wise** 多种粒度 + **INT8 / INT4 / FP8 / FP4**;INT4 还配 BF16 scale/bias。
- **位宽**:INT8 是稳妥默认;要更省可 INT4(配 QAT 防掉点,接 2.2)。
- 实践:**Demo B 量化 embedding 就用 TorchRec 的 Quant 模块做 row-wise INT8**,测体积/延迟/精度三角(卡 O-2)。

### 2.4 INT8 在 GPU 降延迟的三因素 + Tensor Core 精度档(🟡,吸收种子,已核实)

低精度降延迟有**三个独立来源**(B3 考点):
1. **Tensor Core 吞吐(compute)**:Volta+ SM 的专用 MMA 单元,一条指令算小矩阵 `D=A×B+C`,只吃低精度。
2. **内存访问(memory)**:INT8 数据是 FP32 的 1/4,**搬得少** → 治 memory-bound(embedding 量化主要吃这条)。
3. **缓存利用率**:同样 cache 装更多。

**Tensor Core 各精度峰值吞吐(A100,已核实,无 sparsity)**:
```
TF32 156 TFLOPS → FP16 312 TFLOPS → INT8 624 TOPS    （逐档约翻倍）
相对 FP32(走 CUDA Core,非 Tensor Core)约 8× / 16× / 32×
```
> ⚠️ 关键认知:**量化 ≠ 一定快**。① 要有对应精度的 Tensor Core MMA 指令 + kernel 真走到该路径;② **embedding lookup 量化是省带宽(memory),不是 Tensor Core(compute)**——只有 dense GEMM 才吃 Tensor Core;③ Tensor Core 对矩阵维度有**对齐要求(8/16 倍数)**,不对齐 fallback 回 CUDA Core → 白量化(对应卡 **B-3**)。

### 2.5 进阶 PTQ 思路:SmoothQuant / AWQ / GPTQ(⚪ 了解,主要面 LLM)

> recsys 精排 dense 不大,这三者主战场是 LLM;了解思路即可,核心都是**治激活 outlier**。

| 方法 | 类型 | 核心思想 |
|---|---|---|
| **GPTQ** | weight-only | 基于 Hessian 的逐层误差补偿,压到 3-bit 仍准 |
| **AWQ** | weight-only | **激活感知**:保护激活幅度大的少数"重要权重"通道,per-channel 缩放 |
| **SmoothQuant** | weight+activation(W8A8) | 用等价变换把量化难度**从激活迁到权重**("抹平"激活 outlier),让激活也能 INT8 |

> 一句话:**outlier 在激活里**——GPTQ/AWQ 绕开(只量化权重),SmoothQuant 正面处理(迁移到权重后两边都量化)。

### 2.6 工具与工程坑

- **工具**:torchao **PT2E** PTQ/QAT(`prepare_pt2e` → 校准 → `convert_pt2e`),官方教程见 track-b 学习资料。
- ⚠️ 坑汇总(对应实践卡):量化方式选错精度崩(**B-2**)、维度不对齐 fallback / 只省体积不提速(**B-3**)、三角要全测——**体积↓ 不代表延迟↓ 不代表精度可接受**(**O-2**)。

### 2.7 面试答法(30 秒)

> recsys 量化第一杠杆是 **embedding 表**——DLRM **>99% 体积在 embedding**(可达 TB),量化它主要**省带宽治 memory-bound**,而非走 Tensor Core(那是 dense GEMM 的事)。具体做法是 **row-wise(每行一组 scale)INT8**,工具用 TorchRec 的 `QuantEmbeddingBagCollection` + 底层 FBGEMM。方法上 **先 PTQ**(快、8-bit 够;PTQ 还分激活 dynamic/static——static 要校准但全 INT8 最快),掉点厉害或要 INT4 再上 **QAT**(贵但准 ~10%)。代价来自**粒度和对称性**:per-tensor 便宜但 outlier 行会拉崩精度,**row-wise/非对称**更准但有开销——选错就是"精度崩"。还有三个坑:量化 ≠ 一定快(要 Tensor Core MMA 支持 + 维度 8/16 对齐,否则 fallback),所以一定把**体积/延迟/精度三角**全测了再下结论。LLM 那套 SmoothQuant/AWQ/GPTQ 核心是治激活 outlier,recsys dense 不大用得少。

> **本节事实来源**:[PTQ vs QAT(精度/成本权衡)](https://medium.com/better-ml/quantization-aware-training-qat-vs-post-training-quantization-ptq-cd3244f43d9a)、[dynamic vs static 量化](https://pytorch.org/blog/quantization-in-practice/)、[Practical Quantization in PyTorch(per-tensor/channel、对称/非对称)](https://pytorch.org/blog/quantization-in-practice/)、[SmoothQuant/AWQ/GPTQ 对比](https://apxml.com/courses/practical-llm-quantization/chapter-3-advanced-ptq-techniques/comparing-advanced-ptq)、[A100 Tensor Core 各精度吞吐(datasheet)](https://www.nvidia.com/content/dam/en-zz/Solutions/Data-Center/a100/pdf/nvidia-a100-datasheet-us-nvidia-1758950-r4-web.pdf)、[embedding 占 DLRM >99% 体积 + INT4 QAT 8× 压缩(DQRM)](https://arxiv.org/abs/2410.20046)、[TorchRec 量化 EmbeddingBagCollection + FBGEMM row-wise/INT4/INT8](https://pytorch.org/blog/torchrec-fbgemm-1/)。

## 3. 算子融合（operator fusion）

### 3.0 问题定位:pointwise 算子是 memory-bound(🟡)
elementwise 加/乘、sigmoid/sin/cos 这类 **pointwise 算子,瓶颈在访存不在算**。eager 模式**每个 op 起一个 kernel**:从 HBM 读 → 算(往往最快的一步)→ 写回 HBM。一串小算子 = 一串"读-算-写"HBM 往返 + 一串 kernel launch → **memory-bound + overhead-bound**(接 B1)。

### 3.1 核心机制:为什么融合能提速(🟢)

把多个算子**合成一个 kernel**,收两份钱:
1. **省 HBM 往返(治 memory)**:中间结果**留在片上**(寄存器 / L1 / shared memory),**不落 HBM**;数据只**读一次、写一次**,不为每个中间值反复读写。这是大头——pointwise 本就 memory-bound。
2. **省 kernel launch(治 overhead)**:N 个 kernel → 1 个,少 N−1 次 launch 派发。

> 一句话:**融合把"读-算-写 ×N"压成"读-算算算-写 ×1"**,砍的是 HBM 流量和 launch 次数,不是算力。典型给 DL 负载 ~15–30% 延迟改善(⚪)。

### 3.2 vertical vs horizontal fusion(🟡)

| | vertical(纵向) | horizontal(横向) |
|---|---|---|
| 融什么 | **生产者-消费者链**(前一个的输出是后一个的输入):`a→relu→mul` | **并发的独立算子**(共享输入、可并行):多个 head/分支 |
| 收益来源 | 消掉链上中间 buffer | 合并 launch + 共享读 |

### 3.3 Inductor 怎么融(🟡):它是编译的子产物,不是独立工具

- **算子融合不是一个你单独调用的东西,是 TorchInductor 编译时自动做的**(接 §1.4 编译栈第三段)。
- Inductor 的 **Scheduler** 把兼容算子融成一个 **Triton kernel**(GPU)/ C++(CPU),融合判据:**common reads(共享输入 buffer 的算子是候选)** + **能省掉中间 buffer**。
- 能融的类别:**pointwise、reduction、template-based**;Inductor 能把**很多 pointwise + reduction 融进一个 Triton kernel**。
- 所以"做算子融合" ≈ `torch.compile` 跑通且没被 graph break 打断(断图处融合也断,接 §1.3)。

### 3.4 bound 归因(🟢,接 B1)
算子融合**主要治 memory-bound(减 HBM 流量)+ overhead-bound(减 launch)**——它优化的是**访存和派发,不是算力**。所以对小算子多、pointwise 多的负载收益大(recsys 拼装/激活、LayerNorm 这类),对本就 compute-bound 的大 GEMM 收益有限(那是量化+Tensor Core 的活,§2.4)。

### 3.5 工程坑与验证(对应卡 O-4 / B-3)
- **怎么确认融合真生效**(⚪):`TORCH_LOGS=output_code` 看 Inductor 生成的 fused kernel 源码;`torch.profiler` 看 **kernel 数下降 + HBM 流量下降 + 延迟下降**(三者一起才算数)。
- 🪖 **卡 O-4**:未融合 → compile 后 kernel 数↓ / HBM 流量↓ / 延迟↓,这就是融合战绩。
- 🪖 **卡 B-3 的一支**:以为融合了其实没融(被 graph break 打断、或算子间插了 `.cpu()`/打印强制物化)→ kernel 数没降、没提速。

### 3.6 面试答法(30 秒)

> 算子融合治 recsys 常见的 **memory-bound + overhead-bound**:pointwise 算子(激活、加乘、LayerNorm)瓶颈在访存,eager 每个 op 一次"读 HBM-算-写 HBM"+ 一次 launch。融合把多个算子合成一个 kernel,**中间结果留在片上寄存器/L1 不落 HBM,数据只读一次写一次**,还省掉 N−1 次 launch——砍的是访存和派发不是算力。分 vertical(生产者-消费者链)和 horizontal(并发算子)。实现上它**不是独立工具,是 torch.compile / Inductor 编译时自动做的**(Scheduler 按 common reads 融成 Triton/C++ kernel),所以前提是 compile 没被 graph break 打断。验证看 `TORCH_LOGS=output_code` + profiler 的 kernel 数和 HBM 流量是否真降。

> **本节事实来源**:[kernel fusion 减 HBM 往返 / vertical·horizontal](https://www.abhik.ai/articles/kernel-fusion)、[Inductor 融合 pointwise/reduction + eager 每 op 一 kernel(PyTorch 性能调优指南)](https://docs.pytorch.org/tutorials/recipes/recipes/tuning_guide.html)、[TorchInductor Scheduler 融合判据(common reads / 省 buffer)](https://deepwiki.com/pytorch/pytorch/2.2-inductor)。

## 4. 蒸馏（distillation）

### 4.0 问题定位:蒸馏换的是"更小的模型本身"(🟡)
编译/量化/融合都**不改模型结构**(同一个模型跑更快);**蒸馏直接换一个更小的 student 模型**,用大 teacher 指导它训练。所以蒸馏**三种 bound 一起降**(模型小了,算/访存/派发都少),代价是**掉精度**——能不能接受看 student 学得多像 teacher。

### 4.1 核心机制:soft label + 温度 + 暗知识(🟢)

- **hard label**(0/1 真标签)信息少;teacher 的 **soft label(softmax 概率分布 / logits)** 含**暗知识(dark knowledge)**——"这个样本像 A 类 0.7、也有点像 B 类 0.2",类间相似度信息比硬标签丰富。
- **温度 τ**:softmax 除以 τ 再 softmax,**τ 越大分布越软**,越凸显非目标类的暗知识。student 同时学 soft label(蒸馏损失,带 τ)+ hard label(真标签损失)。
- ⚠️ 固定全局 τ 的坑:不同样本 logit 尺度/难度不同,单一 τ 会让有的预测过尖(信息少)、有的过平(丢判别性)。

### 4.2 三类蒸馏(🟡)

| 类 | 对齐什么 |
|---|---|
| **logit 蒸馏** | 直接匹配**输出分布**(最常用) |
| **feature 蒸馏** | 对齐**中间层表示**(student 学 teacher 的隐层) |
| **relation 蒸馏** | 对齐**样本间关系**结构 |

### 4.3 recsys 特殊用法(🟢,这节的重点,面试亮点)

1. **大精排 → 小模型**:把重 teacher 蒸成轻 student 上线,压缩 + 加速(经典模型压缩用法)。
2. **粗排蒸精排**:让粗排 student 学精排 teacher 的打分 → **缓解粗排/精排阶段不一致**(粗排截断掉的好候选,精排本会捞回——蒸馏让粗排"预见"精排偏好)。
3. **🌟 privileged features distillation(特权特征蒸馏)**:**特权特征 = 离线训练有、在线 serving 取不到的特征**(如未来信息、贵特征)。teacher 用全部特征(含特权)训练,蒸给一个**不含特权特征的 student** 上线 serving。这是 recsys 蒸馏区别于 CV/NLP 的独门用法——蒸的不只是"小",还是"用不到的特征的知识"。
4. **ranking distillation**:取 teacher 推荐 list 的 **top-K** 当额外监督,引导 student 给这些 item 更高分。

### 4.4 online vs offline 蒸馏(⚪)
- **offline**(最常用):teacher 先训好、冻结,再蒸 student。
- **online / 联合训练**:teacher 和 student 同时更新、端到端一起训(recsys 排序常用联合训练形态)。

### 4.5 bound 归因与取舍(🟢,接 B1)
- 蒸馏是**唯一"换模型"的杠杆**,所以**三种 bound 一起降**,但**唯一会主动牺牲精度**的(其余三个理论上近似无损)。
- 取舍轴:**student 容量 ↔ 精度 ↔ 延迟**。容量太小学不像 teacher → 掉点;太大 → 没省到。

### 4.6 工程坑(对应卡 O-5 / B-4)
- 🪖 **卡 O-5**:teacher → student,测 **参数量↓ / 延迟↓ / AUC 掉幅可控** 的三者权衡曲线。
- 🪖 **卡 B-4**:蒸馏掉点 / teacher-student 不匹配——**温度 τ 设错、logits 没对齐、student 容量太小** → student AUC 比自训还低;调 τ / 对齐 soft-label / 加容量后回升。

### 4.7 面试答法(30 秒)

> 蒸馏是四杠杆里**唯一换模型本身**的——用大 teacher 的 **soft label(暗知识)** 教小 student,student 同时学 soft label(带温度 τ)和真标签,所以**三种 bound 一起降但会掉精度**。分 logit / feature / relation 蒸馏。recsys 有三个特色用法:① 大精排蒸小模型上线;② **粗排蒸精排缓解阶段不一致**;③ **privileged features distillation**——teacher 用离线才有的特权特征训练,蒸给在线取不到这些特征的 student,这是 recsys 独门。坑主要是温度设错、teacher-student 不匹配、student 容量太小导致掉点,要测 student 容量↔精度↔延迟的权衡曲线。

> **本节事实来源**:[soft label / 温度 / dark knowledge / logit·feature·relation 三类](https://arxiv.org/html/2403.01427v1)、[recsys 蒸馏:粗排蒸精排联合训练 / ranking distillation top-K(综述)](https://www.mdpi.com/2079-9292/14/8/1538)、[privileged features distillation(离线特征→在线 student)](https://arxiv.org/abs/2312.08727)、[online vs offline 蒸馏](https://arxiv.org/html/2408.14678v1)。

## 5. 四杠杆 × bound 收口表（接 B1 三分法）

| 杠杆 | 主治 bound | 改模型结构? | 掉精度? | recsys 关键点 | 主要坑 |
|---|---|---|---|---|---|
| **编译**(compile/export) | overhead(+memory) | 否 | 否 | 减 launch;部署走 AOTInductor | graph break、重编译(变长) |
| **算子融合** | memory + overhead | 否 | 否 | pointwise 留片上不落 HBM | 被 graph break 打断就没融 |
| **量化 embedding** | memory(带宽) | 否 | 微 | **row-wise INT8**,>99% 体积 | 选错粒度精度崩 |
| **量化 dense + Tensor Core** | compute | 否 | 微 | 维度 8/16 对齐才走 Tensor Core | 不对齐 fallback、白量化 |
| **蒸馏** | 三种一起 | **是** | **是** | privileged features / 粗排蒸精排 | 温度、容量、teacher-student 不匹配 |

> 读法:**先用 B1 定位 bound,再查表选杠杆**。memory-bound 先想量化 embedding + 融合;overhead-bound 先想编译 + CUDA Graphs;compute-bound 才是量化 dense + Tensor Core;要更小就蒸馏(接受掉点)。
